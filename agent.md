# GuardedOps Agent Control File

## 1. 项目定位

GuardedOps 是 AI Hackathon 2026《操作系统智能代理》的参赛项目。

项目目标是构建一个“安全对话式 Linux/SSH 运维代理”：

- 用户用自然语言提出 Linux 系统管理需求；
- 系统将自然语言解析为结构化意图；
- 系统基于真实环境信息、任务类型和安全策略生成受控执行计划；
- 执行前进行风险识别、风险预警、范围限制和二次确认；
- 执行层只调用白名单工具，不允许任意 shell；
- 执行后给出自然语言解释、状态验证和审计日志。

GuardedOps 的核心不是“会执行命令”，而是“安全、可解释、可审计地代理执行系统操作”。

---

## 2. 当前已锁定方向

### 2.1 交互形态

采用混合形态：

- Web 为主演示入口；
- CLI 为辅助调试入口；
- Web 和 CLI 共用同一套 parser / policy / planner / executor / audit 核心引擎。

### 2.2 执行模式

必须支持：

- LocalExecutor：本地 Linux 执行；
- SSHExecutor：通过 SSH 远程执行。

### 2.3 Agent 形态

采用“规则优先 + LLM 辅助”的混合方案：

- 规则和代码风控是最终安全边界；
- Prompt 只辅助自然语言理解、解释和总结；
- 不允许让 LLM 直接生成任意 bash 并执行；
- 不允许 arbitrary shell / raw command mode。

---

## 3. 项目硬边界

### 3.1 禁止事项

GuardedOps 严禁实现以下能力：

- 万能 shell 聊天机器人；
- 用户输入自然语言后直接生成 bash 执行；
- raw command mode；
- arbitrary shell execution；
- `shell=True` 执行用户可控字符串；
- 修改 sudoers；
- 修改 sshd_config；
- 修改防火墙规则；
- 创建 sudo/root/admin 用户；
- 删除 root、当前用户或系统用户；
- 批量 chmod/chown；
- 批量删除用户；
- 删除系统核心目录；
- 自动修复关键系统配置；
- 不经确认执行敏感写操作。

### 3.2 能力范围

MVP 阶段只覆盖：

- 磁盘使用情况查询；
- 文件或目录检索；
- 进程查询；
- 端口查询；
- 普通用户创建；
- 普通用户删除；
- 高风险请求识别、拒绝、解释；
- 多轮上下文与连续任务闭环；
- 审计日志与验证材料。

---

## 4. 风控原则

GuardedOps 的风控原则是：

> Prompt 可以解释风险，但不能承担最终安全边界。最终安全边界必须由代码实现。

所有写操作必须经过以下流程：

1. 结构化意图解析；
2. 参数校验；
3. 环境探测；
4. 风险分级；
5. 计划预览；
6. 二次确认；
7. 白名单工具执行；
8. 执行后状态验证；
9. 审计记录。

所有未知写操作默认拒绝。

---

## 5. 风险等级定义

| 风险等级 | 含义 | 默认处理 |
|---|---|---|
| S0 | 只读安全操作 | 可直接执行，但仍记录审计 |
| S1 | 受限变更 | 需要普通确认 |
| S2 | 敏感变更 | 需要强确认 |
| S3 | 禁止执行 | 拒绝并解释 |

示例：

- 磁盘查询：S0
- 文件检索：S0，若范围过大则要求收窄或截断
- 进程查询：S0
- 端口查询：S0
- 创建普通用户：S1
- 删除普通用户：S2
- 删除 `/etc`、`/boot`、`/usr` 等核心目录：S3
- 修改 sudoers、sshd_config：S3
- 创建 sudo/root/admin 用户：S3

## 6. 工具协作规则

### 6.1 ChatGPT 的职责

ChatGPT 用于：

- 项目总控；
- 架构决策；
- 任务拆解；
- Prompt 设计；
- 演示脚本设计；
- 文档润色；
- Codex 输出审查；
- 线程接力总结。

ChatGPT 不直接承担最终代码生成主力。

### 6.2 Codex 的职责

Codex 用于：

- 按任务 ID 实现代码；
- 创建或修改指定文件；
- 编写测试；
- 修复局部 bug；
- 执行小范围重构；
- 生成工程文档草稿。

Codex 每次只允许执行一个明确任务，不允许自行扩展功能范围。

### 6.3 Gemini 的职责

Gemini 用于：

- 安全审查；
- 边界条件审查；
- 演示脚本挑错；
- 文档表达审查；
- 提交前反向评审。

Gemini 不直接改代码，除非明确要求。

## 7. 阶段判断规则

### Phase 0：仓库与总控体系初始化

判断标准：

- 还没有稳定的 repo 结构；
- 管理文档未创建；
- 任务板未建立；
- 提示词体系未建立。

### Phase 1：可运行底座与只读能力

判断标准：

- 开始实现 LocalExecutor / SSHExecutor；
- 开始实现 env_probe；
- 开始实现磁盘、文件、进程、端口只读工具；
- 开始实现基础 CLI / Web 骨架。

### Phase 2：受限写操作与确认闭环

判断标准：

- 开始实现创建普通用户；
- 开始实现删除普通用户；
- 开始实现风险等级、确认语、拒绝规则；
- 开始实现风险卡片和确认按钮。

### Phase 3：多轮上下文与连续任务

判断标准：

- 开始处理“刚才那个用户”“继续查刚才端口”等上下文；
- 开始实现多步计划；
- 开始实现连续任务执行和中断恢复。

### Phase 4：审计、验证、演示材料

判断标准：

- 开始完善 SQLite / JSONL 审计；
- 开始准备 demo scenarios；
- 开始整理自测报告；
- 开始打通录屏脚本。

### Phase 5：交付文档、答辩与最终润色

判断标准：

- 核心能力已可运行；
- 开始准备最终 README、架构说明、Agent 配置、核心 Prompt、工具定义文档；
- 开始进行最终安全审查和提交包整理。

## 8. 线程分流规则

### 8.1 每个线程只做一种任务

建议分流：

- 线程 A：项目总控和任务规划；
- 线程 B：Codex 实现某个任务；
- 线程 C：测试和错误修复；
- 线程 D：文档和演示；
- 线程 E：Gemini 安全审查。

禁止在一个线程中同时进行：

- 大范围架构调整；
- 代码实现；
- 文档重写；
- 安全策略变更；
- 演示脚本改写。

### 8.2 每个 Codex 任务必须绑定 Task ID

所有给 Codex 的任务必须包含：

- Task ID；
- 当前阶段；
- 目标；
- 允许修改文件；
- 禁止修改文件；
- 完成标准；
- 测试要求；
- 不得扩 scope 的说明。

### 8.3 不允许隐式改架构

如果 Codex 认为需要改架构，必须停下并输出：

- 为什么当前架构不够；
- 建议修改点；
- 影响文件；
- 风险；
- 是否需要更新 `decision_log.md`。

未经确认不得自行改。

---

## 9. 状态记录规则

每次完成一个任务后，必须更新：

1. `task_board.md`
   - 更新任务状态；
   - 填写完成时间；
   - 填写输出物；
   - 填写遗留问题。

2. `current_status.md`
   - 更新当前阶段；
   - 更新当前焦点；
   - 更新下一步；
   - 更新阻塞点；
   - 更新最新决策。

3. `decision_log.md`
   - 仅当发生架构、安全、技术选型等关键决策时更新。

4. `validation_matrix.md`
   - 当新增或完成可验证能力时更新。

---

## 10. 接力上下文规则

每次从一个 AI 线程切到另一个 AI 线程，必须使用 `handoff_template.md`。

交接内容必须包含：

- 当前阶段；
- 当前任务 ID；
- 已完成内容；
- 未完成内容；
- 文件变更；
- 风险点；
- 待确认项；
- 下一步建议。

不要把大量历史对话原样粘贴给新线程，应该用结构化摘要交接。

---

## 11. 线程污染防护

为避免线程污染，必须遵守：

- 不在实现线程中重新讨论总方案；
- 不在文档线程中提出未确认的新功能；
- 不在测试线程中随意扩大业务范围；
- 不让 Codex 同时修改多个互相依赖的核心模块；
- 不让任何 AI 基于过期上下文继续开发；
- 每次新线程开始前先读取：
  - `agent.md`
  - `project_context.md`
  - `architecture_constraints.md`
  - `current_status.md`
  - 当前任务提示词
